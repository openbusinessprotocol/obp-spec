# OBP-COMMERCE — BAAB Commerce Protocol

Status: Adopted (PROP-093, 2026-04-07)
Corresponding PROP: PROP-072 (original adoption 2026-04-03), superseded by PROP-093

---

## Overview

OBP-COMMERCE is the specification that describes at the protocol level
the transaction structure BAAB (Business-Agent-Agent-Business),
in which agents are the executing parties of commercial transactions.

Through flow.delegate (OBP-TRUST), agents can act as proxies for organizational will.
OBP-COMMERCE extends that delegation to "commercial transactions with external organizations."

---

## OBP Axes

```
flow.commerce     — flow.G (dynamics, progress of commercial transactions)
resource.contract — resource.T (recording of contracts, consideration, obligations)
context.market    — context.C (the semantic space where transactions are concluded)
```

## 1. Definition of BAAB

BAAB (Business-Agent-Agent-Business) is the transaction structure where agents are the executing parties.

```
B2B (conventional):
  Human (Org A) ─[negotiation, contract, invoice, payment]─ Human (Org B)

BAAB:
  Human (Org A) → Agent(A) ─[flow.commerce]─ Agent(B) → Human (Org B)
                     ↑                               ↑
              authorized by                   authorized by
              Org A via                       Org B via
              flow.delegate                   flow.delegate
```

Humans exist as "approval gates" and "configurers"; agents are the executing parties.
Legal responsibility returns to the organizations (humans) at both ends of the transaction.

Human roles in BAAB:

```
Onboarding: set delegation scope, approval thresholds, payment limits
Normal transactions: agents execute autonomously (no human involvement)
Approval gate: human intervenes based on high-amount, irreversible, exception flags
Post-confirmation: periodically review transaction logs and trust scores
```

### Handling Asymmetric BAAB (One-Sided BAAB)

In addition to "full BAAB" where both agents hold flow.delegate,
asymmetric transaction structures where one party is human (B2AB / BA2B) also occur in practice.

```
Full BAAB: Agent(A) ─ Agent(B)   both sides have flow.delegate
B2AB:      Agent(A) ─ Human(B)   recipient side is human
BA2B:      Human(A)  ─ Agent(B)  ordering side is human
```

OBP-COMMERCE defines its event set with full BAAB as the standard.
The same event specification can be used in B2AB / BA2B,
but when one side lacks flow.delegate, that agent's `delegate_proof` field is omitted.

## 2. flow.commerce Event Specification

flow.commerce is the minimum protocol event set that establishes BAAB.

### Common Fields (Required for All Events)

```
idempotency_key  unique identifier. generated by the firing party. duplicate firing with same key is ignored.
event_at         firing datetime (ISO 8601)
trace_id         WF trace reference (for audit and debug)
```

### Event List

```
work_completed
  Description: Agent completed task/workflow; origin point for consideration billing
  Required fields:
    org_id              identifier of billing organization
    agent_id            identifier of completing agent
    task_ref            reference to task/workflow
    deliverable_summary summary of deliverable (natural language)
    amount_suggested    billing amount calculated by agent (human can verify/change)
    currency            currency code (ISO 4217)
    due_date_expected   expected payment due date (ISO 8601)
    delegate_proof      proof of flow.delegate (OBP-TRUST grant reference)

invoice_created
  Description: Invoice draft has been generated
  Precondition: context.prohibition (PROP-017) query must be complete.
                If query returns prohibition_blocked, do not fire this event.
  Required fields:
    invoice_id          invoice identifier
    from_org            billing organization identifier
    to_org              billed organization identifier
    line_items          array of line items (each element: description, quantity, unit price)
    total_amount        total amount
    currency            currency code
    requires_approval   whether human approval is required (boolean)

invoice_approved
  Description: Invoice has been approved (by human or agent)
  Required fields:
    invoice_id          invoice identifier
    approved_by         type of approving party ('human' or 'agent')
    approver_id         identifier of approving human or agent

payment_completed
  Description: Payment has been completed (notification received from payment infrastructure)
  Required fields:
    invoice_id          invoice identifier
    payment_provider    payment provider identifier
    amount              payment amount
    currency            currency code
    paid_at             payment completion datetime (ISO 8601)

feedback_submitted
  Description: Human has evaluated transaction quality
  Required fields:
    invoice_id          target invoice identifier
    rating              evaluation value (integer 1–5)
    submitted_by        evaluator identifier
  Optional fields:
    comment             free text

transaction_disputed
  Description: Dispute has been filed against a transaction
  Required fields:
    invoice_id          target invoice identifier
    disputed_by         filer identifier
    org_id              filing organization identifier
    reason              reason for dispute (natural language)
    dispute_type        dispute classification
                        Possible values: non_delivery / quality / unauthorized / other

contract_proposed
  Description: Agent has generated a contract draft
  Precondition: same as invoice_created, PROP-017 query must be complete.
  Required fields:
    contract_id         contract identifier
    template_id         identifier of template used
    parties             array of party organization identifiers
    terms_summary       summary of contract terms (natural language)
    requires_human_signature  whether human signature is required (boolean)

contract_signed
  Description: Contract has been concluded
  Required fields:
    contract_id         contract identifier
    signed_by           array of signers (each element: identifier, signature type, datetime)
    effective_date      contract effective date (ISO 8601)

prohibition_blocked
  Description: context.prohibition query result prohibited transaction firing
  Required fields:
    attempted_event     event name that was attempted
    axis                array of violated AXIS (see PROP-017 §1)
    reason              prohibition reason (natural language)
    blocked_by          basis for prohibition (value: 'context.prohibition')
```

## 3. Integration with Trust Score

BAAB transaction history is reflected in OBP trust scores. OBP-COMMERCE defines the following contracts:

```
When payment_completed fires:
  Record transaction completion in trust score of target org_id and agent_id.
  Reflect payment due date adherence (comparison with due_date_expected).

When feedback_submitted fires:
  Reflect quality evaluation based on rating value in trust score.

When transaction_disputed fires:
  Reflect penalty based on dispute_type in trust score.
```

Trust score calculation formulas, coefficients, and weighting are defined by the World Filter (implementing platform).
OBP-COMMERCE defines the contract of "what to reflect"; "how to calculate" is delegated.

Trust score components (flow.commerce related):

```
Number of completed transactions
Payment due date adherence rate
Quality evaluation (rating distribution from feedback_submitted)
Dispute incidence rate (count and type of transaction_disputed)
```

## 4. Integration with context.prohibition

OBP's context.prohibition (PROP-017) is positioned above charter and applied before all flow.*.

Integration rules with flow.commerce:

```
Before firing invoice_created and contract_proposed,
the executing party has a duty to query context.prohibition (PROP-017 AXIS-1–4).

If an AXIS violation is detected:
  → Stop firing the target event
  → Fire the prohibition_blocked event
  → Consider the flow.delegate establishment conditions unmet (see PROP-071)

Only when the query result is CAL-2 (Cleared) can invoice_created / contract_proposed be fired.
```

## 5. Relationships to Other Protocols

```
OBP-ID       → identification of organizations and agents (who)
OBP-TRUST    → proof of delegation via flow.delegate (permission)
OBP-SIG      → signature/agreement attestation (evidence)
OBP-ACT      → integration with context.prohibition (compliance)
OBP-COMMERCE → progression of commercial transaction events (flow)
contract record → data record of contract content (resource)

Complete BAAB transaction = OBP-ID + OBP-TRUST + OBP-COMMERCE + OBP-SIG + contract record
```
